# Broken Pony Club Client Portal

A lightweight, premium-feeling client workspace platform for support delivery, inspired by Linear/ClickUp workflows.

This project is a React + Supabase SaaS app focused on:

- multi-workspace client collaboration,
- task lifecycle management,
- in-app notifications,
- time tracking against prepaid support retainers.

---

## 1) Project Overview

**Broken Pony Club Client Portal** is a multi-tenant project management and support operations interface where each client organization maps to a workspace.

Core outcomes:

- provide clients with transparent visibility into support progress,
- track effort in real time,
- align logged time to retainer hours,
- create a clean, premium portal experience that is easy to use.

## 6) System Architecture

### Frontend stack

- **Build/Runtime:** Vite + React + TypeScript
- **Routing:** `react-router-dom`
- **Data/cache:** `@tanstack/react-query`
- **Backend client:** `@supabase/supabase-js`
- **UI:** Tailwind + shadcn-style component primitives + lucide icons

### Architectural pattern

- Route-driven SPA with protected routes.
- Modular API layer in `src/api/*` encapsulating Supabase queries.
- Workspace-scoped query keys for cache isolation.
- UI composed from reusable `components/ui/*` primitives.

### Auth behavior

- Production path: Supabase password-only auth (no OTP/magic link on public login).
- Dev path: local bypass (`isDevBypassEnabled`) for localhost review without credentials.

## 7) Database Schema Overview

Expected Supabase objects used by the app:

### Tables

- `workspaces(id, name, created_at)`
- `workspace_users(workspace_id, user_id, role)`
- `tasks(id, workspace_id, title, description, status, due_date, assignee_user_id, created_by, created_at, updated_at)`
- `comments(id, task_id, user_id, body, created_at)`
- `task_activity(id, task_id, type, payload jsonb, created_at)`
- `time_entries(id, workspace_id, task_id, user_id, started_at, ended_at, duration_seconds, created_at)`
- `support_buckets(id, workspace_id, period_start, period_end, hours_allocated numeric, hours_used_cached numeric)`
- `notifications(id, workspace_id, user_id, type, payload jsonb, read_at, created_at)`

### Views

- `my_workspaces(id, name)`
- `workspace_support_summary(workspace_id, hours_allocated, hours_used, hours_remaining)`

### RPC functions

- `start_task_timer(p_workspace_id uuid, p_task_id uuid)`
- `stop_task_timer(p_workspace_id uuid, p_task_id uuid)`

### Fallback behavior

If views/RPC are missing, client-side fallbacks in `src/api/*` use base tables.

### Security expectations

- RLS should be enabled and enforced in Supabase.
- Client code assumes workspace/user access checks are handled by policies.

## 10) Environment Variables

Create `.env.local` from `.env.example`:

```bash
cp .env.example .env.local
```

Required keys:

```env
VITE_SUPABASE_URL=https://YOUR-PROJECT-REF.supabase.co
VITE_SUPABASE_ANON_KEY=YOUR_SUPABASE_ANON_KEY
```

Optional dev convenience:

```env
VITE_DEV_BYPASS_AUTH=true
```

### Notes

- If Supabase keys are missing, the app can still run in local dev with bypass mode.
- For production/staging, always provide valid Supabase credentials.

### Edge function environment variables

For the invite-client edge function, set:

```env
SUPABASE_URL=https://YOUR-PROJECT-REF.supabase.co
SUPABASE_SERVICE_ROLE_KEY=YOUR_SERVICE_ROLE_KEY
APP_BASE_URL=https://your-app-url.com (optional, fallback to request origin)
```

### Demo mode

When `VITE_DEV_BYPASS_AUTH=true` **or** Supabase keys are missing, the app runs in demo mode with in-memory data and no live Supabase calls.

Review paths:

- `/workspaces`
- `/w/demo/dashboard`
- `/w/demo/tasks`
- `/w/demo/time`
- `/w/demo/reports`
- `/w/demo/settings`

## 11) Production Deployment Instructions

### Build artifact

- Static SPA output in `dist/` via `npm run build`.

### Deployment targets

- Vercel, Netlify, Cloudflare Pages, or any static host with SPA fallback.

### Required production configuration

1. Set env vars:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
2. Ensure app routes rewrite to `index.html` (SPA history fallback).
3. Confirm Supabase Auth redirect URLs include your deployed domain.
4. Verify RLS policies are active before go-live.

### Suggested release checklist

- `npm run lint`

## 12) Coding Standards

- TypeScript-first; avoid `any` unless unavoidable.
- Keep Supabase calls inside `src/api/*` modules.
- Use React Query for server state and cache invalidation.
- Use workspace-scoped query keys (`src/lib/queryKeys.ts`).
- Keep UI consistent with Tailwind theme tokens (no arbitrary brand color drift).
- Prefer reusable primitives in `src/components/ui/*`.
- Keep components focused and composable.
- Maintain naming clarity (`Page`, `Provider`, `Widget`, `Table`, `Drawer`).

## Inviting clients

Workspace admins can invite clients from the Workspace Settings page. The invite email contains a link to set a password and join the workspace. After setting their password, clients can log in using email and password.

The invite flow uses a Supabase Edge Function (`invite-client`) with the following environment variables:

- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `APP_BASE_URL` (optional)

See `supabase/functions/invite-client/index.ts` for implementation details.
