## 1) Project Overview

**Broken Pony Club Client Portal** is a multi-tenant project management and support operations interface where each client organization maps to a workspace.

Core outcomes:

- provide clients with transparent visibility into support progress,
- track effort in real time,
- align logged time to retainer hours,
- create a clean, premium portal experience that is easy to use.

## 6) System Architecture

### Frontend stack

- **Build/Runtime:** Vite + React + TypeScript
- **Routing:** `react-router-dom`
- **Data/cache:** `@tanstack/react-query`
- **Backend client:** `@supabase/supabase-js`
- **UI:** Tailwind + shadcn-style component primitives + lucide icons

### Architectural pattern

- Route-driven SPA with protected routes.
- Modular API layer in `src/api/*` encapsulating Supabase queries.
- Workspace-scoped query keys for cache isolation.
- UI composed from reusable `components/ui/*` primitives.
- Centralized Edge Function authentication via `invokeAuthedFunction` helper with automatic session refresh.

### Auth behavior

- Production path: Supabase password-only auth.
- Dev path: local bypass (`isDevBypassEnabled`) for localhost review without credentials.
- Edge Function calls: Centralized through `invokeAuthedFunction` helper that ensures valid session and attaches Authorization header with automatic refresh if token is expiring within 30 seconds.

If views/RPC are missing, client-side fallbacks in `src/api/*` use base tables.

### Storage Buckets

- `avatars` - Stores user profile images
  - **Required Setup**: Create this bucket manually in Supabase Storage
  - Access: Public read, Authenticated write
  - RLS Policy Example:

    ```sql
    -- Allow authenticated users to upload their own avatar
    CREATE POLICY "Users can upload own avatar"
    ON storage.objects FOR INSERT
    TO authenticated
    WITH CHECK (bucket_id = 'avatars' AND (storage.fspath(name))[1] = auth.uid()::text);

    -- Allow public read access
    CREATE POLICY "Public avatar access"
    ON storage.objects FOR SELECT
    TO public
    USING (bucket_id = 'avatars');
    ```

### Fallback behavior

If views/RPC are missing, client-side fallbacks in `src/api/*` use base tables.

### Security expectations

- RLS should be enabled and enforced in Supabase.
- Client code assumes workspace/user access checks are handled by policies.

```env
SUPABASE_ANON_KEY=YOUR_SUPABASE_ANON_KEY
APP_BASE_URL=https://your-app-url.com (optional, fallback to request origin)
```

**Important Edge Function Configuration:**

After deploying Edge Functions to Supabase, you must disable the "Verify JWT with legacy secret" setting in the Supabase Dashboard for each function:

1. Go to: `https://supabase.com/dashboard/project/{project-ref}/functions/{function-name}/details`
2. Find "Verify JWT with legacy secret" toggle
3. Turn it **OFF**
4. Click "Save changes"

This allows the Edge Functions to handle JWT verification in code (recommended approach) rather than Supabase automatically requiring the anon key as the Authorization header.

When `VITE_DEV_BYPASS_AUTH=true` **or** Supabase keys are missing, the app runs in demo mode with in-memory data and no live Supabase calls.

3. Confirm Supabase Auth redirect URLs include your deployed domain.
4. Verify RLS policies are active before go-live.
5. Create required database function in Supabase SQL Editor:

```sql
-- Drop the existing function if it exists
DROP FUNCTION IF EXISTS get_workspace_users_with_emails(uuid);

-- Create the function with avatar_url support
CREATE OR REPLACE FUNCTION get_workspace_users_with_emails(workspace_id_param UUID)
RETURNS TABLE (
  user_id UUID,
  email TEXT,
  avatar_url TEXT
)
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
  RETURN QUERY
  SELECT
    wu.user_id,
    u.email::TEXT,
    (u.raw_user_meta_data->>'avatar_url')::TEXT
  FROM workspace_users wu
  INNER JOIN auth.users u ON u.id = wu.user_id
  WHERE wu.workspace_id = workspace_id_param
  ORDER BY u.email;
END;
$$;
```

6. Configure Edge Functions (disable "Verify JWT with legacy secret" for each deployed function).

## Inviting clients

Workspace admins can invite clients from the Workspace Settings page. The invite email contains a link to set a password and join the workspace. After setting their password, clients can log in using email and password.

Admins can edit the workspace name directly in settings. The invite flow uses only the client email and sends the workspace name automatically.

Note: The invite-client edge function must be deployed to Supabase for invites to work. Local POST to /functions/v1/invite-client will 404 unless running in Supabase local dev.
